# llm-wiki Init 強化：結構化知識庫 Profile

## 概述

強化 `llm-wiki` 的 init 流程，透過更豐富的提問建立結構化的「知識庫 Profile」寫入 `CLAUDE.md`。後續的 ingest / query / lint 操作在執行前先讀取 profile，依據設定調整行為。re-run init 時進入更新模式，可局部修改既有設定。

## 動機

目前 init 只問四件事（主題、目錄、來源格式、語言），寫入 `CLAUDE.md` 的內容過於單薄，導致後續操作缺乏個性化指導。使用者對於摘要風格、引用策略、矛盾處理等偏好無處表達，LLM 只能猜測。

## 設計

### 1. Init 問題清單

init 時依序詢問以下問題，每題附範例答案。使用者可以直接選範例、自己寫、或跳過。

#### 基本資訊（現有，保留）

1. **主題／領域** — 知識庫關於什麼？
2. **專案目錄** — 建在哪裡？
3. **語言** — wiki 內容使用的語言

#### 目的與使用場景（新增）

4. **建立目的** — 這個知識庫主要用來做什麼？
   - 範例：「研究特定領域的學術文獻」「產品競品分析與市場調研」「個人學習筆記整理」「團隊知識共享」「專案技術決策記錄」

5. **預期使用者** — 誰會查閱這個知識庫？
   - 範例：「只有我自己」「我和幾位同事」「整個團隊」「未來的我，幫助回憶脈絡」

6. **主要來源類型** — 通常會匯入什麼？（可複選）
   - 範例：「學術論文 PDF」「技術部落格文章」「會議記錄」「書籍章節」「訪談逐字稿」「產品文件」「新聞報導」

#### 整理邏輯與偏好（新增）

7. **組織方式** — 你希望知識怎麼被組織？
   - 範例：「按主題分類，像百科全書」「按時間線排列，追蹤演變」「按因果關係連結，理解為什麼」「按專案／產品分組」

8. **摘要風格** — 你偏好什麼樣的摘要？
   - 範例：「精簡扼要，每頁 200-300 字」「詳盡完整，保留關鍵細節」「結構化條列，方便快速掃描」

9. **原文引用** — 需要保留來源的原文引用嗎？
   - 範例：「需要，重要論點要附原文」「不需要，用自己的話改寫就好」「只保留關鍵數據和定義的原文」

#### 品質標準（新增）

10. **矛盾處理** — 不同來源說法衝突時怎麼辦？
    - 範例：「並列呈現，標註各自來源」「以最新的為準，舊的標記過時」「依證據強度排序」

11. **信心標注** — 需要標注資訊的可信度嗎？
    - 範例：「不需要，太繁瑣」「需要，用 ⚠️ 標記未經驗證的資訊」「需要，標注證據等級（強／中／弱）」

#### 領域特有結構（新增）

12. **自訂欄位** — 你的領域有沒有每個條目都該記錄的特殊資訊？
    - 範例：「技術：版本相容性、效能指標」「醫學：證據等級、樣本量」「商業：市場規模、競爭格局」「法律：適用法規、判例編號」「沒有特別的」

### 2. CLAUDE.md Profile 結構

init 根據問答結果，在 `CLAUDE.md` 中產生以下結構。分兩層：高層原則（指導精神）與具體指引（直接改變操作行為的規則）。

```markdown
# LLM Wiki Schema

## 知識庫概述
- 主題：{topic}
- 目的：{purpose}
- 預期使用者：{audience}
- 語言：{language}

## 高層原則
- 組織邏輯：{organization_style}
- 摘要風格：{summary_style}
- 知識庫調性：{由 LLM 根據以上回答綜合判斷，一句話描述}

## 具體指引

### 來源處理
- 主要來源類型：{source_types}
- 原文引用：{quote_policy}

### 頁面結構
- 自訂欄位：{custom_fields，寫成具體的 frontmatter 或區塊規範}

### 品質控制
- 矛盾處理策略：{contradiction_policy}
- 信心標注：{confidence_policy}

## 頁面類型
- **來源摘要**（source）：每個匯入來源一頁
- **實體頁**（entity）：人物、組織、產品、地點、事件
- **概念頁**（concept）：理論、方法、框架
- **綜整頁**（synthesis）：跨來源的分析與整合
- **比較頁**（comparison）：並排比較
{如有自訂頁面類型，列在此處}

## 規則
- raw/ 中的檔案不可修改
- 每次操作後更新 index.md 和 log.md
- 使用 [[wikilink]] 格式交叉引用
- 新資料與舊主張矛盾時，依矛盾處理策略處理
- 每個 wiki 頁面包含 YAML frontmatter
```

**設計要點：**

- 「高層原則」寫的是精神——LLM 遇到模糊情況時依此判斷
- 「具體指引」寫的是規則——直接決定頁面要不要有 Key Quotes 區塊、frontmatter 要不要加 `confidence` 欄位等
- 「知識庫調性」由 LLM 綜合所有回答後自行歸納，例如：「嚴謹的學術研究知識庫，重視證據與溯源」或「輕量的個人學習筆記，重視快速查閱」

### 3. Init 更新模式

當使用者對已初始化的知識庫再次執行 `/llm-wiki init` 時，進入更新模式。

#### 偵測邏輯
- 檢查 `wiki/index.md` 和 `CLAUDE.md` 是否存在
- 若存在 → 進入更新模式
- 若不存在 → 正常初始化流程

#### 更新模式流程

1. **讀取現有 CLAUDE.md**，解析目前的 profile 設定
2. **向使用者展示目前設定摘要**，例如：
   ```
   目前知識庫設定：
   1. 主題：LLM 技術研究
   2. 目的：追蹤學術論文與技術進展
   3. 摘要風格：結構化條列
   4. 原文引用：只保留關鍵數據和定義
   5. 矛盾處理：並列呈現，標註各自來源
   ...

   要修改哪些項目？（輸入編號，或「全部重新設定」）
   ```
3. **只重新詢問使用者選擇的項目**，其餘保留不動
4. **更新 CLAUDE.md**，保留未變動的內容
5. **追加 log.md**，記錄 `update-schema` 操作與變更了哪些欄位

#### 安全機制
- 更新前不刪除舊 `CLAUDE.md`，而是原地修改對應區塊
- 如果 `CLAUDE.md` 中有使用者手動加的其他內容（非 wiki schema 區塊），保留不動

### 4. 其他操作參考 CLAUDE.md 的方式

#### Ingest — 新增前置步驟

在現有「前置檢查」之後、「識別來源」之前，插入：

> **讀取 CLAUDE.md profile**
> - 確認摘要風格 → 控制摘要長度與格式
> - 確認原文引用策略 → 決定是否產生 Key Quotes 區塊
> - 確認自訂欄位 → 在 frontmatter 和頁面內容中加入對應欄位
> - 確認矛盾處理策略 → 遇到衝突時依策略處理
> - 確認信心標注 → 決定是否加入可信度標記

#### Query — 新增前置步驟

在現有「讀取 index.md」之前，插入：

> **讀取 CLAUDE.md profile**
> - 確認知識庫目的與使用者 → 調整回答的深度與語氣
> - 確認組織邏輯 → 影響搜尋相關頁面的策略
> - 確認摘要風格 → 回答風格與 profile 一致

#### Lint — 新增檢查項目

在現有七項檢查之外，新增：

> **Profile 一致性檢查**
> - 頁面是否符合 CLAUDE.md 中的具體指引？（例如設定「不保留原文引用」但某些頁面有 Key Quotes）
> - 自訂欄位是否在所有頁面中一致存在？
> - 摘要長度是否符合風格設定？

這些不一致可能是因為 profile 更新後舊頁面還沒跟上，lint 會標出來讓使用者決定是否批次修正。

## 改動範圍

| 檔案 | 改動內容 |
|------|----------|
| `SKILL.md` — Init 區段 | 擴充問題清單（12 題含範例）、新增更新模式邏輯、更新 CLAUDE.md 模板 |
| `SKILL.md` — Ingest 區段 | 新增「讀取 CLAUDE.md profile」前置步驟 |
| `SKILL.md` — Query 區段 | 新增「讀取 CLAUDE.md profile」前置步驟 |
| `SKILL.md` — Lint 區段 | 新增「Profile 一致性檢查」項目 |

`references/conventions.md` 不需要改動——它定義的是頁面格式的通用規範，profile 中的具體指引會在此基礎上做增減。
