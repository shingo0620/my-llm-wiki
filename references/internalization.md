# 內化機制細節指引

本文件規範三段式思考環節（Anchor / Defend / Challenge / Compose）的執行細節，以及暫存草稿格式。SKILL.md 的 ingest 與 query 章節引用此檔，避免重複描述。

設計參考：`docs/superpowers/specs/2026-05-18-internalization-design.md`

## 共通原則

1. **提問完整性**：所有對使用者的提問必須 inline 完整描述主張內容、結論、紅隊論點。不可用「這個」「該主張」「上述」等代名詞依賴使用者記憶
2. **profile 驅動**：執行前讀 CLAUDE.md 的 `## 內化機制` 區塊，依設定調整題數、字數預期、是否啟用 Challenge
3. **產出寫入位置**：`## 我的觀點` 寫在頁面最末段——來源頁則在 `## 與 Wiki 的關聯` 之後、綜整頁則在 `## 結論` 之後、比較頁則在 `## 結論與建議` 之後

## Anchor（定錨）

### 目的
快速定位使用者對核心主張的立場類別，作為後續 Defend 與 Challenge 的起點。

### 流程
1. 萃取 2-3 個核心主張（ingest）或 1-2 個（query）
2. 每個主張獨立一道選擇題
3. 提問必須 inline 完整描述主張，含具體數據、條件、範圍

### 主張萃取啟發式（依來源類型自適應）

| 來源類型 | 萃取重點 |
|---------|---------|
| 學術論文 / 技術報告 | 核心方法 + 主要實驗結論 + 適用範圍 |
| 評論 / 觀點文章 | 中心論點 + 關鍵推論 + 結論立場 |
| 訪談 / Podcast 逐字稿 | 受訪者主要主張 + 反映的世界觀 |
| 比較性文章 | 各方立場 + 作者偏好 |
| 新聞 / 報導 | 事件詮釋 + 因果歸因 |
| YouTube 演講 | 講者核心觀點 + 預測 / 建議 |

### 選擇題模板

```
主張 X：「<完整描述主張內容，含數據、條件、範圍——50-150 字>」

你的反應最接近：
A. 完全同意——<為什麼這是合理的同意路徑>
B. 部分同意——<最常見的部分同意理由>
C. 不同意 / 存疑——<最常見的反對理由>
D. 中立——<為什麼可能還無法表態>
```

### 選項生成原則

- A 不是套話「同意」，要寫出「為什麼會同意的具體理由」
- B 預期是最有思考密度的選項，要寫出「同意 X 但 Y 存疑」的具體 X 與 Y
- C 要寫出最強反方的具體論點，避免使用者選了 C 卻不知為什麼反對
- D 給「資訊不足」的退路，但避免變成逃避選項——若 D 過於誘人，調整 A-C 涵蓋度

### A 選項說明與 Defend 階段的關係

A 選項 label 中的「<為什麼會同意的具體理由>」是 **LLM 預先提供的同意路徑說明**，給使用者判斷「A 是否確實符合我的立場」用——**不是**使用者的 Defend 論述。

即使使用者選了 A，Defend 階段仍會執行（依「全選 A 的情境」規則：挑最具爭議的一題追問「最強同意理由」）。LLM 應主動向使用者說明這個邏輯，避免誤解：

> 你選了主張 X = A。即使選 A，我會挑最具爭議的一題追問你「最強同意理由是什麼？」——A 選項裡的理由是我的預設說明，不能取代你自己的論述。

## Defend（辯護）

### 目的
逼使用者把選擇題的立場類別轉成自己的論述，避免「點完選項就過」的儀式化。

### 觸發規則
- 對 Anchor 中選 B、C、D 的主張：要求 50-150 字（ingest）或 50 字內（query）解釋理由
- 對全選 A 的情境：LLM 從中挑「最具爭議的一個主張」追問「最讓你同意的具體理由是什麼？」——避免無腦同意。爭議性判斷啟發式（依序）：
  1. wiki 中存在對立觀點或矛盾紀錄的主張
  2. 來源中作者本人有讓步、附帶條件、或承認反對意見的主張
  3. 若皆無，挑論述最強（佐證最豐富）的主張，因為它最承擔結論

### 提問模板

提問時必須 inline 重述使用者表態的主張內容，不可用「主張 X」「該主張」等代名詞——這呼應共通原則第 1 條。

```
你對「<inline 完整重述該主張內容，30-80 字，含關鍵數據或條件>」選了 <B/C/D>。請說明：<具體要釐清的判準>？
你會拿什麼樣的<證據/數字/案例>當基準？
```

### 字數彈性
- profile 中設定的字數是預期值，不是硬性下限
- 使用者明顯敷衍（< 字數預期的 50%）時，LLM 追問一次具體點：「能舉個例子嗎？」「具體的判準是什麼？」——只追問一次，不糾纏

## Challenge（挑戰）

### 觸發條件
- ingest：profile 中 `ingest 啟用挑戰階段` 為 true 時啟用（預設）
- query：永遠跳過（query 簡化版無此階段）

### 紅隊論點來源優先順序

1. **wiki 中現有的對立觀點**：搜尋既有 wiki 頁面（特別是 entity、concept 頁），看是否有與當前使用者立場相左的記錄。若有，引用具體頁面與段落
2. **使用者過去的立場（自我紅隊）**：若使用者過去對相關主張有不同立場，引用作為「你過去這樣想」的內部挑戰
3. **LLM 從訓練知識生成的反方論點**：以上都無時，由 LLM 自行構造反方論點

優先 1 與 2，因為它們把挑戰錨定在 wiki 內部，產生跨頁面互相挑戰的網絡。

### 紅隊論點品質要求

- **完整描述**：論點必須 inline 自包含，不可只說「反方會反對」就要使用者自己想
- **有實質內容**：不是修辭，要有具體推論、數據、案例
- **針對使用者選的立場**：根據 Anchor 與 Defend 的內容客製，避免出泛用紅隊

### 提問模板

```
你說「<inline 完整重述使用者 Defend 論點的核心，20-50 字，不依賴前文上下文>」。但反方會這樣說：

「<完整紅隊論點，含推論鏈、具體證據或案例，100-300 字>」

你怎麼回應？
```

### 回應後的處理
- 使用者回應後直接進 Compose 階段，不繼續多輪辯論（避免拖延）
- 若使用者明確表示「想再深一輪」，可繼續一輪後強制收斂到 Compose

## Compose（整理）

### 目的
把 Anchor + Defend + Challenge 的對話整理為結構化的 `## 我的觀點` 區塊，寫入來源頁/綜整頁底部。

### 寫入位置
- 來源頁、綜整頁、比較頁的最末段
- 來源頁在 `## 與 Wiki 的關聯` 之後、綜整頁在 `## 結論` 之後、比較頁在 `## 結論與建議` 之後
- 用 H2 `## 我的觀點`，下面用 H3 dated subheading（`### YYYY-MM-DD`）

### 模板（ingest 完整版）

````markdown
## 我的觀點

### YYYY-MM-DD

**對核心主張的立場：**
- 主張 1（<簡短描述>）：<完全同意/部分同意/存疑/中立>（<A/B/C/D>）— <一行理由>
- 主張 2（<簡短描述>）：<選擇>（<選項代號>）— <一行理由>
- 主張 3（<簡短描述>）：<選擇>（<選項代號>）— <一行理由>

**我的論述：**
<整合使用者在 Defend 階段的回答，50-150 字>

**對紅隊質疑的回應：**
<整合使用者在 Challenge 階段的回應，附帶紅隊論點要旨>

**未解問題：**
- <使用者明示或暗示的疑問>
- <Challenge 階段未被解決的張力>

**標籤：** #stance:<整體立場> #strength:<信心強度>
````

### 模板（query 簡化版）

````markdown
## 我的觀點

### YYYY-MM-DD（query 產出）

**對結論的立場：**
- 結論（<簡短描述>）：<選擇>（<選項代號>）— <一行理由>

**我的論述：**
<整合使用者 Defend 回答，50 字內>

**標籤：** #stance:<整體立場>

（query 簡化版不輸出 `#strength:`，因無 Challenge 階段。）
````

### 多題立場彙整邏輯

當 Anchor 多題立場不一致時，整體 `#stance:` 標籤的取值規則：

1. **取最保守端**：分歧時靠 C 端（skeptical）方向。例如 1A + 2B + 3C → `#stance:skeptical`（不是平均）
2. **多 B 但無 C**：1B + 2B + 3A → `#stance:partial-agree`
3. **全同意**：1A + 2A + 3A → `#stance:agree`
4. **全存疑或反對**：1C + 2C + 3D → `#stance:skeptical`（neutral 只在多數 D 時用）

LLM 在 Compose 階段應在「**對核心主張的立場：**」段下方**加一行**說明取值依據（例如「整體標 `#stance:partial-agree`，因主張 2 存疑但主張 1、3 不至於完全否定」），避免標籤與分項立場看起來矛盾。

### 標籤命名空間

（本檔與 `references/conventions.md` 同步維護標籤定義。任何修改應同步兩處。）

- `#stance:agree` — 對應 A
- `#stance:partial-agree` — 對應 B
- `#stance:skeptical` — 對應 C
- `#stance:neutral` — 對應 D
- （`#strength:*` 標籤僅適用於 ingest 完整版三段式。query 簡化版無 Challenge 階段，不輸出 `#strength:`。）
- `#strength:strong` — 經 Defend + Challenge 仍維持立場
- `#strength:moderate` — Defend 有合理論述但 Challenge 部分動搖
- `#strength:weak` — Challenge 後立場明顯偏移

## 暫存草稿（`.pending/`）

### 觸發
使用者在三段式任何階段明確說「先暫停」「等等回來」「中斷」「下次再做」等語意時觸發。LLM 不主動推測——只接受明示。

### 檔案位置
`wiki/.pending/<source-or-synthesis-filename>.md`

例：
- ingest 中斷：`wiki/.pending/attention-is-all-you-need.md`
- query 中斷：`wiki/.pending/transformer-vs-rnn.md`

### 檔案格式

（以下範例假設使用者於 2026-05-18 14:30 暫停 ingest 操作；LLM 實際使用時應填入當下日期時間。）

```markdown
---
target_page: wiki/attention-is-all-you-need.md
trigger_op: ingest
paused_at_stage: defend  # anchor / defend / challenge / compose
paused_date: 2026-05-18  # 範例值，實際填入當下日期
---

## Anchor 進度
<已答過的選擇題與答案>

## Defend 進度
<已答過的開放題與答案>

## Challenge 進度
（尚未進入）

## 暫存原因
使用者於 2026-05-18 14:30 說「先暫停，等等回來」  # 範例值，實際填入當下日期時間與說法
```

### 恢復流程
1. 下次跑 `/llm-wiki ingest` 或 `/llm-wiki query` 時，skill 開頭掃描 `.pending/`
2. 若不為空，列出未完成的草稿（檔名、target_page、paused_at_stage、paused_date）
3. 要求使用者選擇：(a) 恢復某份草稿（從 paused_at_stage 開始繼續）（b) 捨棄某份草稿（c) 都先放著，開始新操作（**但 lint 會持續提示**）
4. 恢復後完成思考環節，正常流程寫入 target_page 並刪除 `.pending/` 中對應檔

### 過期偵測
lint 偵測 `paused_date` 超過 profile 設定天數（預設 14）的草稿，提示使用者恢復或捨棄。
