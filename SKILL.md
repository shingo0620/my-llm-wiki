# LLM Wiki — 知識庫管理

LLM Wiki 讓 LLM 逐步建立並維護一個持久的 wiki——一組結構化、彼此連結的 Markdown 檔案。不同於 RAG 每次查詢都重新擷取，wiki 將知識整理一次、持續更新，產生複利效果。

人類負責：挑選來源、引導分析、提出問題。
LLM 負責：摘要、交叉引用、歸檔、整理與維護。

## 可用指令

| 指令 | 說明 |
|------|------|
| `/llm-wiki init` | 初始化新的知識庫專案結構 |
| `/llm-wiki ingest` | 匯入新來源，建立與更新 wiki 頁面 |
| `/llm-wiki query` | 查詢知識庫，整合多頁回答 |
| `/llm-wiki lint` | 健康檢查，找出問題並建議改善 |

若使用者未指定操作，根據對話上下文判斷：
- 「初始化」、「建立知識庫」、「開一個新 wiki」→ **init**
- 「匯入」、「處理這篇」、「加入來源」、「ingest」、提供 URL 或 YouTube 連結 → **ingest**
- 提出問題、「查一下」、「關於 X 的資訊」→ **query**
- 「檢查」、「清理」、「維護」、「lint」→ **lint**

## 專案架構

```
<project-root>/
├── raw/                  # 原始資料（不可變，LLM 只讀不改）
│   └── assets/           # 圖片與附件
├── wiki/                 # LLM 生成與維護的 Markdown 頁面
│   ├── index.md          # 內容索引（按類別組織）
│   └── log.md            # 操作日誌（僅追加）
└── CLAUDE.md             # Wiki schema 設定（定義結構與慣例）
```

## 內化機制

ingest 與 query 不只整理資料——當 CLAUDE.md profile 啟用內化機制時，會在流程中插入**硬性思考關卡**，逼使用者對來源主張或結論表態、辯護、面對紅隊挑戰，並將結果以 `## 我的觀點` 區塊累加在 wiki 頁面底部。後續操作主動引用既有觀點，讓「我的判斷」與來源主張並列為 wiki 第一公民。

預設啟用。若想關閉，跑 `/llm-wiki init` 進更新模式調整。

設計與細節見：
- 觸發點與三段式骨架：本檔的 `Ingest` 與 `Query` 段
- 執行細節（題目生成、紅隊原則、Compose 模板、`.pending/` 格式）：[`references/internalization.md`](references/internalization.md)
- 頁面格式與 reflect log：[`references/conventions.md`](references/conventions.md)

## Init — 初始化知識庫

### 0. 偵測模式

檢查當前目錄是否已有 `wiki/index.md` 和 `CLAUDE.md`：
- **都不存在** → 正常初始化流程（步驟 1 起）
- **都存在** → 直接跳至本節末尾的「更新模式」區段，**不執行步驟 1-6**
- **部分存在** → 提示使用者狀態異常，確認要重新初始化還是修復

### 1. 確認設定

依序詢問以下問題。每題提供範例答案供參考，使用者可直接選範例、自己寫、或跳過。若對話中已明確提供某項資訊，跳過該題。

#### 基本資訊

1. **主題／領域** — 知識庫關於什麼？
   - 範例：「機器學習論文研究」「SaaS 產品競品分析」「個人投資學習」「團隊技術決策記錄」

2. **專案目錄** — 建在哪裡？（預設：當前目錄）

3. **語言** — wiki 內容使用的語言
   - 範例：「繁體中文」「English」「日本語」

#### 目的與使用場景

4. **建立目的** — 這個知識庫主要用來做什麼？
   - 範例：「研究特定領域的學術文獻」「產品競品分析與市場調研」「個人學習筆記整理」「團隊知識共享」「專案技術決策記錄」

5. **預期使用者** — 誰會查閱這個知識庫？
   - 範例：「只有我自己」「我和幾位同事」「整個團隊」「未來的我，幫助回憶脈絡」

6. **主要來源類型** — 通常會匯入什麼？（可複選）
   - 範例：「學術論文 PDF」「技術部落格文章」「會議記錄」「書籍章節」「訪談逐字稿」「產品文件」「新聞報導」「網頁 URL」「YouTube 影片」

#### 整理邏輯與偏好

7. **組織方式** — 你希望知識怎麼被組織？
   - 範例：「按主題分類，像百科全書」「按時間線排列，追蹤演變」「按因果關係連結，理解為什麼」「按專案／產品分組」

8. **摘要風格** — 你偏好什麼樣的摘要？
   - 範例：「精簡扼要，每頁 200-300 字」「詳盡完整，保留關鍵細節」「結構化條列，方便快速掃描」

9. **原文引用** — 需要保留來源的原文引用嗎？
   - 範例：「需要，重要論點要附原文」「不需要，用自己的話改寫就好」「只保留關鍵數據和定義的原文」

#### 品質標準

10. **矛盾處理** — 不同來源說法衝突時怎麼辦？
    - 範例：「並列呈現，標註各自來源」「以最新的為準，舊的標記過時」「依證據強度排序」

11. **信心標注** — 需要標注資訊的可信度嗎？
    - 範例：「不需要，太繁瑣」「需要，用 ⚠️ 標記未經驗證的資訊」「需要，標注證據等級（強／中／弱）」

#### 領域特有結構

12. **自訂欄位** — 你的領域有沒有每個條目都該記錄的特殊資訊？
    - 範例：「技術：版本相容性、效能指標」「醫學：證據等級、樣本量」「商業：市場規模、競爭格局」「法律：適用法規、判例編號」「沒有特別的」

#### 內化機制

13. **內化深度** — 你希望 wiki 強迫你做多少思考？
    - 範例：
      - 「最強：每次 ingest 都進完整三段思考、query 立場性結論也進思考」（對應全開）
      - 「中等：ingest 啟用三段但關掉挑戰階段、query 不觸發」
      - 「最輕：完全關閉內化機制，只整理不思考」

14. **觀點演化檢查** — 你想要 lint 主動指出你新舊觀點的衝突嗎？
    - 範例：
      - 「想，矛盾本身就是學習機會」
      - 「不用，太頻繁提醒會疲乏」

### 2. 建立結構

建立 `raw/`、`raw/assets/`、`wiki/` 目錄。

### 3. 建立 wiki/index.md

```markdown
# 索引

> 本索引由 LLM 自動維護。每次匯入或更新時同步更新。

## 來源摘要

（尚無來源）

## 實體

（尚無實體頁面）

## 概念

（尚無概念頁面）

## 綜整與分析

（尚無綜整頁面）

## 比較

（尚無比較頁面）
```

### 4. 建立 wiki/log.md

```markdown
# 日誌

> 僅追加記錄。格式：`## [YYYY-MM-DD] 操作 | 標題`

## [YYYY-MM-DD] init | 知識庫初始化
- 主題：{topic}
- 目的：{purpose}
- 語言：{language}
- Profile 設定：已寫入 CLAUDE.md
```

建立 log.md 時，以步驟 1 收集到的實際值取代 `{topic}`、`{purpose}`、`{language}` 等佔位符。

### 5. 建立 CLAUDE.md（知識庫 Profile）

在專案根目錄建立或更新 CLAUDE.md。Profile 分為兩層：高層原則（指導精神，LLM 遇到模糊情況時依此判斷）與具體指引（直接改變操作行為的規則）。

````markdown
# LLM Wiki Schema

## 知識庫概述
- 主題：{topic}
- 目的：{purpose}
- 預期使用者：{audience}
- 語言：{language}

## 高層原則
- 組織邏輯：{organization_style}
- 摘要風格：{summary_style}
- 知識庫調性：{由 LLM 根據所有回答綜合判斷，一句話描述，例如「嚴謹的學術研究知識庫，重視證據與溯源」或「輕量的個人學習筆記，重視快速查閱」}

## 具體指引

### 來源處理
- 主要來源類型：{source_types}
- 原文引用：{quote_policy}

### 頁面結構
- 自訂欄位：{custom_fields}
  （若有，寫成具體的 frontmatter 欄位或頁面區塊規範；若無，整行省略）

### 品質控制
- 矛盾處理策略：{contradiction_policy}
- 信心標注：{confidence_policy}

## 頁面類型
- **來源摘要**（source）：每個匯入來源一頁
- **實體頁**（entity）：人物、組織、產品、地點、事件
- **概念頁**（concept）：理論、方法、框架
- **綜整頁**（synthesis）：跨來源的分析與整合
- **比較頁**（comparison）：並排比較
（如有自訂頁面類型，在此以相同格式追加）

## 內化機制（internalization）

- 啟用：{internalization_enabled}
- ingest 啟用挑戰階段：{ingest_challenge_enabled}
- ingest Anchor 題數：{ingest_anchor_count}（範圍 2-3）
- ingest Defend 字數預期：50-150
- query 觸發策略：寬鬆（寧可多觸發、避免漏抓）
- query Anchor 題數：1-2（query 自動簡化，無 Challenge 階段）
- query Defend 字數預期：50 字內
- 暫存過期天數：14
- lint 偵測新舊觀點衝突：{lint_stance_conflict_enabled}

## 規則
- raw/ 中的檔案不可修改
- 每次操作後更新 index.md 和 log.md
- 使用 [[wikilink]] 格式交叉引用
- 新資料與舊主張矛盾時，依矛盾處理策略處理
- 每個 wiki 頁面包含 YAML frontmatter
````

根據使用者的回答調整 profile 內容。不要照搬範本——理解使用者的需求後產出符合其目的與領域的指引。「知識庫調性」由 LLM 綜合所有回答後自行歸納一句話。

**內化機制參數對應第 13、14 題**：
- 第 13 題「最強」→ `internalization_enabled: true`、`ingest_challenge_enabled: true`、`ingest_anchor_count: 3`
- 第 13 題「中等」→ `internalization_enabled: true`、`ingest_challenge_enabled: false`、`ingest_anchor_count: 2`
- 第 13 題「最輕」→ `internalization_enabled: false`（其他欄位用預設值寫入，方便日後切回 true）
- 第 14 題「想」→ `lint_stance_conflict_enabled: true`
- 第 14 題「不用」→ `lint_stance_conflict_enabled: false`

### 6. 告知後續步驟

- 將來源放入 `raw/`
- `/llm-wiki ingest` 匯入來源
- `/llm-wiki query` 查詢知識庫
- `/llm-wiki lint` 健康檢查
- 推薦用 Obsidian 開啟專案目錄即時瀏覽

### 更新模式

當步驟 0 偵測到已初始化的知識庫時，進入此流程：

1. **讀取現有 CLAUDE.md**，解析目前的 profile 設定
2. **展示目前設定摘要**，以編號列表呈現所有設定項，例如：
   ```
   目前知識庫設定：
   1. 主題：LLM 技術研究
   2. 目的：追蹤學術論文與技術進展
   3. 摘要風格：結構化條列
   4. 原文引用：只保留關鍵數據和定義
   5. 矛盾處理：並列呈現，標註各自來源
   ...

   要修改哪些項目？（輸入編號，或「全部重新設定」）
   ```
3. **只重新詢問使用者選擇的項目**，附帶範例答案，其餘保留不動
4. **更新 CLAUDE.md**，原地修改對應區塊，保留未變動的內容
5. **追加 log.md**，記錄 `update-schema` 操作與變更了哪些欄位

#### 內化機制的舊 wiki 相容性

若偵測到 CLAUDE.md 不存在 `## 內化機制` 區塊（升級前的舊 wiki），在展示目前設定摘要時**主動提示**：

````
偵測到你的 CLAUDE.md 沒有內化機制設定。內化機制是新功能，會在
ingest 與 query 時加入硬性思考關卡逼出你的觀點。要現在設定嗎？
(y) 是，回答第 13、14 題
(n) 否，跳過——所有內化機制欄位用預設值（全部啟用）寫入 CLAUDE.md
(d) 否，永久停用——`啟用：false` 寫入 CLAUDE.md，可日後再開
````

選 (n) 與 (d) 的差別：(n) 仍會觸發思考關卡，只是用預設參數；(d) 完全不觸發。使用者選 (y) 時走標準的兩題流程。

#### 安全機制
- 不刪除舊 CLAUDE.md，而是原地修改 `# LLM Wiki Schema` 區塊內的對應內容
- CLAUDE.md 中若有使用者手動加的其他內容（非 `# LLM Wiki Schema` 區塊），保留不動

## Ingest — 匯入來源

### 前置檢查

確認 `wiki/index.md` 和 `wiki/log.md` 存在。若不存在，引導使用者先跑 init。

### 讀取 CLAUDE.md Profile

閱讀 CLAUDE.md 中的知識庫 profile，確認以下設定並在後續步驟中遵守：
- **摘要風格** → 控制摘要長度與格式
- **原文引用策略** → 決定是否產生 Key Quotes 區塊
- **自訂欄位** → 在 frontmatter 和頁面內容中加入對應欄位
- **矛盾處理策略** → 遇到衝突時依策略處理
- **信心標注** → 決定是否加入可信度標記
- **內化機制設定** → 決定是否在步驟 8.5 啟用思考關卡，以及 Anchor 題數、Defend 字數預期、Challenge 是否啟用

### 流程

#### 1. 識別來源並重述既有觀點

**識別來源**：確認要處理的來源（檔案 / URL / YouTube 三種形式，詳見下方 a/b/c 子段）。

**重述既有觀點**：來源確認後、進入步驟 2 之前，**搜尋 wiki 中與來源主題相關的既有 `## 我的觀點` 內容**。若有命中，在步驟 8.5 的 Anchor 階段**先 inline 重述既有立場再出題**，讓新證據與舊立場直接對話。

範例（ingest 新論文時觸發）：

```
你正在 ingest「GPT-3: Language Models are Few-Shot Learners」。

提醒：你在 [[attention-is-all-you-need]]（2026-05-18）對「self-attention
通用性」標 #stance:skeptical，寫道：「translation 成功不代表能推廣
到其他序列任務」。

現在這篇來源主張：「在 175B 參數規模下，純 Transformer 架構展現出
few-shot learning 等湧現能力，跨數十種任務（翻譯、QA、算術、文字生成）
無需 fine-tune 即可運作。」

這個證據對你之前的存疑立場是：
A. 推翻——通用性已被證實
B. 部分支持——但 few-shot 不等於跨模態
C. 不影響——few-shot 仍在語言模態內，存疑點是視覺/時序等
D. 我想重新看一下之前的觀點再決定
```

選 D 時 inline 完整貼出 2026-05-18 的觀點段供使用者重讀後再選。

**識別來源（含 a/b/c 三種形式）**：

**a. 檔案** — 來源已在 `raw/` 中，直接進入步驟 2。

**b. URL（網頁）** — 使用者提供一般網頁連結時：
1. 找到腳本路徑並擷取網頁內容存入 `raw/`：
   ```bash
   # 偵測 skill 安裝位置
   SKILL_SCRIPTS=$(find ~/.claude/skills .claude/skills -path "*/llm-wiki/scripts/fetch-url.sh" 2>/dev/null | head -1 | xargs dirname 2>/dev/null)
   "${SKILL_SCRIPTS}/fetch-url.sh" "<URL>" "./raw"
   ```
   腳本會自動：透過 markdown.new API 擷取、以網頁標題的 kebab-case 命名、在檔案開頭加上 `source-url`/`title`/`fetched` metadata
2. 腳本成功時會印出檔案路徑，用該路徑繼續步驟 2

**c. YouTube URL** — 使用者提供 YouTube 影片連結時：
1. 找到腳本路徑並取得逐字稿存入 `raw/`：
   ```bash
   # 偵測 skill 安裝位置（若上方已偵測過則跳過）
   SKILL_SCRIPTS=$(find ~/.claude/skills .claude/skills -path "*/llm-wiki/scripts/fetch-youtube.sh" 2>/dev/null | head -1 | xargs dirname 2>/dev/null)
   "${SKILL_SCRIPTS}/fetch-youtube.sh" "<YouTube URL>" "./raw"
   ```
   腳本會自動：透過 `youtube-transcript-api` 取得逐字稿（首次執行自動安裝）、用影片標題命名（`yt-` 前綴）、加上時間戳與 metadata
2. 腳本成功時會印出檔案路徑，用該路徑繼續步驟 2

**錯誤處理：**
- **腳本路徑找不到**：若 `SKILL_SCRIPTS` 為空（`find` 無結果），告知使用者 skill 可能未正確安裝，建議重新安裝（`npx skills add shingo0620/my-llm-wiki -g -y`）
- **腳本執行失敗**（非零退出碼）：向使用者顯示錯誤訊息（腳本會輸出到 stderr），常見原因包含網路問題、API 限流、YouTube 影片無字幕等。不要靜默忽略，讓使用者決定是否重試或改用其他方式取得來源
- **重複來源**：執行前先檢查 `raw/` 是否已有相同檔名的檔案。若存在，詢問使用者要跳過、覆蓋、還是以新檔名匯入（如 `title-2.md`）
- **大型來源**：若來源超過 5000 行（大型 PDF、長時間影片逐字稿），在步驟 3 先告知使用者來源規模，建議分段重點摘要而非嘗試完整處理

擷取完成後，告知使用者已存入 `raw/` 的檔案路徑，再繼續步驟 2。

#### 2. 閱讀來源
完整讀取來源內容。若來源含圖片，先讀文字再另外查看被引用的圖片。圖片檔案存入 `raw/assets/`，並在 wiki 頁面中以相對路徑引用（`../raw/assets/filename.png`）。

#### 3. 與使用者討論重點
摘要關鍵發現（3-5 點），詢問：
- 哪些重點要強調？
- 有沒有要忽略的部分？
- 與已知內容有哪些關聯？

這一步很重要——使用者的引導讓 wiki 反映他們的視角，而不只是資料的平鋪直述。

#### 4. 建立來源摘要頁
在 `wiki/` 建立摘要頁。格式見 `references/conventions.md`。

#### 5. 更新實體頁
來源中的人物、組織、產品等——建立新頁或更新既有頁面。

#### 6. 更新概念頁
來源涉及的概念、理論、方法——建立或更新。

#### 7. 標註矛盾
若新資料與既有 wiki 主張衝突，在相關頁面明確標註。這是功能，不是 bug——矛盾的存在本身就是有價值的資訊。

#### 8. 維護交叉引用
使用 `[[wikilink]]` 連結相關頁面。主動在新舊頁面之間建立雙向連結。

#### 8.5 思考關卡（內化機制）

> **硬性語意**：本步驟未完成時，步驟 9（更新 index.md）與步驟 10（追加 log.md）**不執行**。這代表此次 ingest 等同未發生——index 上看不到、log 也沒紀錄。使用者下次會看到「raw/ 有檔案但 wiki 沒對應頁面」訊號，自然被拉回完成思考。

若 CLAUDE.md 的 `## 內化機制` 區塊 `啟用：false`，跳過本步直接進步驟 9。否則：

1. **Anchor**：從步驟 4 產出的來源頁中萃取核心主張（題數依 profile，預設 3（範圍 2-3））。每題完整描述主張內容（不用「這個」「該主張」等代名詞），出 A/B/C/D 選擇題給使用者
2. **Defend**：對使用者選非 A 的主張追問理由（50-150 字依 profile）；若全選 A，挑最具爭議的一題追問「最強同意理由」（Defend 永遠執行，僅字數彈性依 profile）
3. **Challenge**（若 profile 啟用）：LLM 從反方視角提一個有實質內容的紅隊論點，使用者回應一段。紅隊論點優先從 wiki 既有對立觀點或使用者過去立場取材，無則 LLM 自行構造
4. **Compose**：整理三階段問答為結構化 `## 我的觀點` 區塊，dated subheading，寫入步驟 4 產出的來源頁底部

**暫存草稿**：使用者在任一階段主動明確說「先暫停」「等等回來」等（LLM 不主動推測），將進度寫入 `wiki/.pending/<filename>.md`，**步驟 9、10 仍不執行**——直到下次恢復完成。

詳細執行指引、選擇題模板、紅隊論點生成原則、Compose 模板、`.pending/` 檔案格式，見 [`references/internalization.md`](references/internalization.md)。

#### 9. 更新 index.md
在適當類別下新增或更新條目。

#### 10. 追加 log.md
記錄 ingest 操作、影響的頁面列表。若步驟 8.5 執行了，**同時**追加 `reflect` op（兩筆紀錄同日先後排列）。reflect op 格式見 `references/conventions.md` 的「### reflect op 格式」段。

### 原則
- **不修改 `raw/` 中的檔案**——它們是不可變的真實來源
- 單一來源可能影響 10-15 個 wiki 頁面，這是正常的
- 偏好一次匯入一個來源，與使用者持續互動
- 批次匯入可行，但監督程度較低
- **內化是 ingest 的一部分**——若 profile 啟用，未完成思考關卡 = ingest 未完成。不要繞過、不要分批做、不要承諾「之後再思考」（避免硬性語意失效）

## Query — 查詢知識庫

### 前置檢查

確認 `wiki/index.md` 和 `CLAUDE.md` 存在。若不存在，引導使用者先跑 init。

### 流程

#### 0. 讀取 CLAUDE.md Profile

閱讀 CLAUDE.md 中的知識庫 profile，確認以下設定：
- **知識庫目的與使用者** → 調整回答的深度與語氣
- **組織邏輯** → 影響搜尋相關頁面的策略
- **摘要風格** → 回答風格與 profile 一致
- **內化機制設定** → 決定 query 是否在步驟 5（詢問存入 wiki）之後觸發簡化版思考關卡，以及觸發策略（寬鬆 / 嚴格）

#### 1. 讀取 index.md
掃描索引找出與問題相關的頁面。

#### 2. 讀取相關頁面
深入閱讀找到的 wiki 頁面。在中等規模（約 100 來源、數百頁）下，索引 + 閱讀的方式意外地很有效。

#### 3. 整合回答
綜合多頁資訊，附帶引用（標明來源頁面）。

**主動引用使用者觀點**：除了 wiki 中的來源主張，**主動檢索使用者既有 `## 我的觀點` 區塊**中與問題相關的立場，整合進回答。回答口吻從「根據 wiki」轉變為「根據 wiki + 你的累積立場」，個人化。

範例：

```
問題：「Transformer 在所有序列任務上都優於 RNN 嗎？」

從 [[attention-is-all-you-need]]、[[transformer-architecture]]、
[[gpt-3]] 三頁綜合：在 NLP 任務上是、長序列且記憶體受限的場景
RNN 仍有利基。

但注意你的立場演化（從 wiki 中 #stance: 標籤聚合而來）：
- 2026-05-18 對「self-attention 通用性」標存疑
- 2026-08-12 對 GPT-3 跨任務 few-shot 表態為「部分支持」

結論建議：對 NLP 內部任務接受、對跨模態保留——這與你的累積立場一致。
```

#### 4. 選擇呈現格式
根據問題性質：
- 一般問題 → Markdown 段落
- 比較 → 表格
- 時間序列 → 時間線
- 複雜分析 → 結構化報告
- 簡報需求 → Marp 格式

#### 5. 詢問是否存入 wiki
好的回答值得保存。比較、分析、新發現的關聯——這些不應消失在聊天紀錄裡。若使用者同意，先依下方流程確認思考關卡是否觸發，完成後存為綜整頁或比較頁、更新 index、追加 log（含 reflect op）。

**存入時的硬性思考關卡**：若 CLAUDE.md 的 `## 內化機制` 啟用、且 LLM 判斷產出為「立場性結論」（非單純事實檢索），在寫入綜整/比較頁**之前**跑簡化版三段式：

- **Anchor**：1-2 題對結論表態（題數依 profile）
- **Defend**：1 個開放題，50 字內（依 profile）
- **Compose**：寫入綜整/比較頁底部 `## 我的觀點` 區塊
- （**無 Challenge 階段**——query 比 ingest 輕，且基底事實已在來源頁內化過）

> **硬性語意**：思考關卡未完成時，綜整/比較頁不寫入、index 不更新、log 不追加（**含 query 與 reflect 兩筆**）——等同此次 query 結論未被保存。

**觸發判斷**：採寬鬆策略——當答案性質介於「事實」與「立場」之間時偏向觸發，避免漏抓。明確跳過的情況：
- 純資料查詢（誰、何時、什麼數據、定義是什麼）
- 使用者同意存入但表明「只是參考、不需要立場記錄」

詳細觸發啟發式、Anchor / Defend 提問模板、Compose query 簡化版模板，見 [`references/internalization.md`](references/internalization.md)。

### 回答品質要求
- 引用具體 wiki 頁面，不要泛泛而談
- 標明結論有多少來源支撐
- 指出知識缺口：「目前 wiki 中這方面資料不足」
- 建議可進一步探索的方向

## Lint — 健康檢查

### 前置檢查

確認 `wiki/index.md`、`wiki/log.md` 和 `CLAUDE.md` 存在。若不存在，引導使用者先跑 init。

### 檢查項目

1. **矛盾**：不同頁面之間的主張衝突
2. **過時資訊**：已被新來源推翻但未更新的舊主張
3. **孤立頁面**：沒有任何反向連結的頁面
4. **缺漏實體**：被多處提及但沒有獨立頁面的重要概念
5. **缺少交叉引用**：應該互相連結但沒有的頁面
6. **資料缺口**：可透過新來源補充的領域
7. **建議問題**：值得進一步探索的新問題與新來源
8. **Profile 一致性**：頁面是否符合 CLAUDE.md 中的具體指引？包含：
   - 設定「不保留原文引用」但某些頁面有 Key Quotes 區塊
   - 自訂欄位是否在所有適用頁面中一致存在
   - 摘要長度是否符合風格設定
   - 這些不一致可能因 profile 更新後舊頁面未跟上，標出讓使用者決定是否批次修正
9. **未思考的來源**：`raw/` 有原始檔但 `wiki/` 對應的來源頁缺失，或來源頁存在但**沒有** `## 我的觀點` 區塊。這是硬性關卡語意的破口（正常 ingest 流程不會產生此狀態，通常源於手動建立頁面或 skill 升級前的舊頁），嚴重度高。建議：跑一次 `/llm-wiki ingest <檔名>` 補完思考
10. **新舊觀點衝突**（僅在 profile `lint 偵測新舊觀點衝突：true` 時啟用）：同一頁面的 `## 我的觀點` 區塊中，兩個 dated subheading 對**同一主張**的 `#stance:` 標籤不一致。把矛盾當學習機會而非錯誤——lint 不自動修，呈現舊觀點與新觀點的完整內容、給三選一觸發演化說明（看了什麼/紅隊論點/兩者皆有），由使用者選擇是否寫入「### 演化說明」子段
11. **過期暫存草稿**：`wiki/.pending/` 中 `paused_date` 超過 profile 設定天數（預設 14）的草稿。嚴重度低。建議：恢復思考完成或捨棄

### 輸出格式

以清單呈現發現，依嚴重程度排序。每項包含：
- 問題描述
- 涉及的頁面
- 建議的修正方式

等使用者確認後再執行修正——不要自動改東西。

### 新舊觀點衝突的特殊處理（檢查項 10）

不只標出衝突，要 inline 完整呈現舊觀點與新觀點的內容，讓使用者不需要回頭翻來源頁就能判斷推進軌跡。範例：

```
你對來源「Attention Is All You Need」中的主張 1（Transformer 革命性
勝過 SOTA）立場變了：

2026-01-15：完全同意（A）— 你寫道：「Transformer 是純粹的架構勝利，
BLEU 提升只是附帶。」

2026-05-18：部分同意（B）— 你寫道：「BLEU +2.1 在飽和區的邊際難度
被低估，但我仍對通用性存疑。」

是哪一個推動了改變？
(a) 看了 [[scaling-laws]] 後修正計算成本框架
(b) 紅隊論點實際說服了你
(c) 兩者皆有 / 其他原因

要把這個演化寫成 `### 演化說明` 子段，留下推進軌跡嗎？
```

使用者選 (a)、(b) 或 (c) 並填寫具體原因後，寫入較新 dated subheading 下方的「#### 演化說明」子段（格式見 `references/conventions.md`）。

### 修正流程

使用者批准後：
1. 執行修正
2. 更新 index.md
3. 追加 log.md 記錄 lint 操作與修正內容

**例外**：檢查項 10（新舊觀點衝突）的「修正」是使用者寫演化說明，不是 LLM 改改觀點。LLM 不應主動移除或修改舊觀點段落。

## 快速參考

### 頁面 Frontmatter

```yaml
---
title: 頁面標題
type: source | entity | concept | synthesis | comparison
sources: [來源1.md, 來源2.md]
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
---
```

### 交叉引用格式

- `[[頁面名稱]]`
- `[[頁面名稱|顯示文字]]`
- `[[頁面名稱#標題]]`

### Index 條目格式

```markdown
- [[頁面名稱]] — 一行摘要（來源數：N）
```

### Log 條目格式

```markdown
## [YYYY-MM-DD] 操作類型 | 標題
- 細節
- 影響頁面：[[page1]], [[page2]]
```

完整頁面格式慣例見 `references/conventions.md`。
