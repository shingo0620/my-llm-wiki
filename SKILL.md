# LLM Wiki — 知識庫管理

LLM Wiki 讓 LLM 逐步建立並維護一個持久的 wiki——一組結構化、彼此連結的 Markdown 檔案。不同於 RAG 每次查詢都重新擷取，wiki 將知識整理一次、持續更新，產生複利效果。

人類負責：挑選來源、引導分析、提出問題。
LLM 負責：摘要、交叉引用、歸檔、整理與維護。

## 可用指令

| 指令 | 說明 |
|------|------|
| `/llm-wiki init` | 初始化新的知識庫專案結構 |
| `/llm-wiki ingest` | 匯入新來源，建立與更新 wiki 頁面 |
| `/llm-wiki query` | 查詢知識庫，整合多頁回答 |
| `/llm-wiki lint` | 健康檢查，找出問題並建議改善 |

若使用者未指定操作，根據對話上下文判斷：
- 「初始化」、「建立知識庫」、「開一個新 wiki」→ **init**
- 「匯入」、「處理這篇」、「加入來源」、「ingest」、提供 URL 或 YouTube 連結 → **ingest**
- 提出問題、「查一下」、「關於 X 的資訊」→ **query**
- 「檢查」、「清理」、「維護」、「lint」→ **lint**

## 專案架構

```
<project-root>/
├── raw/                  # 原始資料（不可變，LLM 只讀不改）
│   └── assets/           # 圖片與附件
├── wiki/                 # LLM 生成與維護的 Markdown 頁面
│   ├── index.md          # 內容索引（按類別組織）
│   └── log.md            # 操作日誌（僅追加）
└── CLAUDE.md             # Wiki schema 設定（定義結構與慣例）
```

## Init — 初始化知識庫

### 0. 偵測模式

檢查當前目錄是否已有 `wiki/index.md` 和 `CLAUDE.md`：
- **都不存在** → 正常初始化流程（步驟 1 起）
- **都存在** → 直接跳至本節末尾的「更新模式」區段，**不執行步驟 1-6**
- **部分存在** → 提示使用者狀態異常，確認要重新初始化還是修復

### 1. 確認設定

依序詢問以下問題。每題提供範例答案供參考，使用者可直接選範例、自己寫、或跳過。若對話中已明確提供某項資訊，跳過該題。

#### 基本資訊

1. **主題／領域** — 知識庫關於什麼？
   - 範例：「機器學習論文研究」「SaaS 產品競品分析」「個人投資學習」「團隊技術決策記錄」

2. **專案目錄** — 建在哪裡？（預設：當前目錄）

3. **語言** — wiki 內容使用的語言
   - 範例：「繁體中文」「English」「日本語」

#### 目的與使用場景

4. **建立目的** — 這個知識庫主要用來做什麼？
   - 範例：「研究特定領域的學術文獻」「產品競品分析與市場調研」「個人學習筆記整理」「團隊知識共享」「專案技術決策記錄」

5. **預期使用者** — 誰會查閱這個知識庫？
   - 範例：「只有我自己」「我和幾位同事」「整個團隊」「未來的我，幫助回憶脈絡」

6. **主要來源類型** — 通常會匯入什麼？（可複選）
   - 範例：「學術論文 PDF」「技術部落格文章」「會議記錄」「書籍章節」「訪談逐字稿」「產品文件」「新聞報導」「網頁 URL」「YouTube 影片」

#### 整理邏輯與偏好

7. **組織方式** — 你希望知識怎麼被組織？
   - 範例：「按主題分類，像百科全書」「按時間線排列，追蹤演變」「按因果關係連結，理解為什麼」「按專案／產品分組」

8. **摘要風格** — 你偏好什麼樣的摘要？
   - 範例：「精簡扼要，每頁 200-300 字」「詳盡完整，保留關鍵細節」「結構化條列，方便快速掃描」

9. **原文引用** — 需要保留來源的原文引用嗎？
   - 範例：「需要，重要論點要附原文」「不需要，用自己的話改寫就好」「只保留關鍵數據和定義的原文」

#### 品質標準

10. **矛盾處理** — 不同來源說法衝突時怎麼辦？
    - 範例：「並列呈現，標註各自來源」「以最新的為準，舊的標記過時」「依證據強度排序」

11. **信心標注** — 需要標注資訊的可信度嗎？
    - 範例：「不需要，太繁瑣」「需要，用 ⚠️ 標記未經驗證的資訊」「需要，標注證據等級（強／中／弱）」

#### 領域特有結構

12. **自訂欄位** — 你的領域有沒有每個條目都該記錄的特殊資訊？
    - 範例：「技術：版本相容性、效能指標」「醫學：證據等級、樣本量」「商業：市場規模、競爭格局」「法律：適用法規、判例編號」「沒有特別的」

### 2. 建立結構

建立 `raw/`、`raw/assets/`、`wiki/` 目錄。

### 3. 建立 wiki/index.md

```markdown
# 索引

> 本索引由 LLM 自動維護。每次匯入或更新時同步更新。

## 來源摘要

（尚無來源）

## 實體

（尚無實體頁面）

## 概念

（尚無概念頁面）

## 綜整與分析

（尚無綜整頁面）
```

### 4. 建立 wiki/log.md

```markdown
# 日誌

> 僅追加記錄。格式：`## [YYYY-MM-DD] 操作 | 標題`

## [YYYY-MM-DD] init | 知識庫初始化
- 主題：{topic}
- 目的：{purpose}
- 語言：{language}
- Profile 設定：已寫入 CLAUDE.md
```

建立 log.md 時，以步驟 1 收集到的實際值取代 `{topic}`、`{purpose}`、`{language}` 等佔位符。

### 5. 建立 CLAUDE.md（知識庫 Profile）

在專案根目錄建立或更新 CLAUDE.md。Profile 分為兩層：高層原則（指導精神，LLM 遇到模糊情況時依此判斷）與具體指引（直接改變操作行為的規則）。

````markdown
# LLM Wiki Schema

## 知識庫概述
- 主題：{topic}
- 目的：{purpose}
- 預期使用者：{audience}
- 語言：{language}

## 高層原則
- 組織邏輯：{organization_style}
- 摘要風格：{summary_style}
- 知識庫調性：{由 LLM 根據所有回答綜合判斷，一句話描述，例如「嚴謹的學術研究知識庫，重視證據與溯源」或「輕量的個人學習筆記，重視快速查閱」}

## 具體指引

### 來源處理
- 主要來源類型：{source_types}
- 原文引用：{quote_policy}

### 頁面結構
- 自訂欄位：{custom_fields}
  （若有，寫成具體的 frontmatter 欄位或頁面區塊規範；若無，整行省略）

### 品質控制
- 矛盾處理策略：{contradiction_policy}
- 信心標注：{confidence_policy}

## 頁面類型
- **來源摘要**（source）：每個匯入來源一頁
- **實體頁**（entity）：人物、組織、產品、地點、事件
- **概念頁**（concept）：理論、方法、框架
- **綜整頁**（synthesis）：跨來源的分析與整合
- **比較頁**（comparison）：並排比較
（如有自訂頁面類型，在此以相同格式追加）

## 規則
- raw/ 中的檔案不可修改
- 每次操作後更新 index.md 和 log.md
- 使用 [[wikilink]] 格式交叉引用
- 新資料與舊主張矛盾時，依矛盾處理策略處理
- 每個 wiki 頁面包含 YAML frontmatter
````

根據使用者的回答調整 profile 內容。不要照搬範本——理解使用者的需求後產出符合其目的與領域的指引。「知識庫調性」由 LLM 綜合所有回答後自行歸納一句話。

### 6. 告知後續步驟

- 將來源放入 `raw/`
- `/llm-wiki ingest` 匯入來源
- `/llm-wiki query` 查詢知識庫
- `/llm-wiki lint` 健康檢查
- 推薦用 Obsidian 開啟專案目錄即時瀏覽

### 更新模式

當步驟 0 偵測到已初始化的知識庫時，進入此流程：

1. **讀取現有 CLAUDE.md**，解析目前的 profile 設定
2. **展示目前設定摘要**，以編號列表呈現所有設定項，例如：
   ```
   目前知識庫設定：
   1. 主題：LLM 技術研究
   2. 目的：追蹤學術論文與技術進展
   3. 摘要風格：結構化條列
   4. 原文引用：只保留關鍵數據和定義
   5. 矛盾處理：並列呈現，標註各自來源
   ...

   要修改哪些項目？（輸入編號，或「全部重新設定」）
   ```
3. **只重新詢問使用者選擇的項目**，附帶範例答案，其餘保留不動
4. **更新 CLAUDE.md**，原地修改對應區塊，保留未變動的內容
5. **追加 log.md**，記錄 `update-schema` 操作與變更了哪些欄位

#### 安全機制
- 不刪除舊 CLAUDE.md，而是原地修改 `# LLM Wiki Schema` 區塊內的對應內容
- CLAUDE.md 中若有使用者手動加的其他內容（非 `# LLM Wiki Schema` 區塊），保留不動

## Ingest — 匯入來源

### 前置檢查

確認 `wiki/index.md` 和 `wiki/log.md` 存在。若不存在，引導使用者先跑 init。

### 讀取 CLAUDE.md Profile

閱讀 CLAUDE.md 中的知識庫 profile，確認以下設定並在後續步驟中遵守：
- **摘要風格** → 控制摘要長度與格式
- **原文引用策略** → 決定是否產生 Key Quotes 區塊
- **自訂欄位** → 在 frontmatter 和頁面內容中加入對應欄位
- **矛盾處理策略** → 遇到衝突時依策略處理
- **信心標注** → 決定是否加入可信度標記

### 流程

#### 1. 識別來源

確認要處理的來源。來源可以是以下三種形式：

**a. 檔案** — 來源已在 `raw/` 中，直接進入步驟 2。

**b. URL（網頁）** — 使用者提供一般網頁連結時：
1. 找到腳本路徑並擷取網頁內容存入 `raw/`：
   ```bash
   # 偵測 skill 安裝位置
   SKILL_SCRIPTS=$(find ~/.claude/skills .claude/skills -path "*/llm-wiki/scripts/fetch-url.sh" 2>/dev/null | head -1 | xargs dirname 2>/dev/null)
   "${SKILL_SCRIPTS}/fetch-url.sh" "<URL>" "./raw"
   ```
   腳本會自動：透過 markdown.new API 擷取、以網頁標題的 kebab-case 命名、在檔案開頭加上 `source-url`/`title`/`fetched` metadata
2. 腳本成功時會印出檔案路徑，用該路徑繼續步驟 2

**c. YouTube URL** — 使用者提供 YouTube 影片連結時：
1. 找到腳本路徑並取得逐字稿存入 `raw/`：
   ```bash
   # 偵測 skill 安裝位置（若上方已偵測過則跳過）
   SKILL_SCRIPTS=$(find ~/.claude/skills .claude/skills -path "*/llm-wiki/scripts/fetch-youtube.sh" 2>/dev/null | head -1 | xargs dirname 2>/dev/null)
   "${SKILL_SCRIPTS}/fetch-youtube.sh" "<YouTube URL>" "./raw"
   ```
   腳本會自動：透過 `youtube-transcript-api` 取得逐字稿（首次執行自動安裝）、用影片標題命名（`yt-` 前綴）、加上時間戳與 metadata
2. 腳本成功時會印出檔案路徑，用該路徑繼續步驟 2

擷取完成後，告知使用者已存入 `raw/` 的檔案路徑，再繼續步驟 2。

#### 2. 閱讀來源
完整讀取來源內容。若來源含圖片，先讀文字再另外查看被引用的圖片。

#### 3. 與使用者討論重點
摘要關鍵發現（3-5 點），詢問：
- 哪些重點要強調？
- 有沒有要忽略的部分？
- 與已知內容有哪些關聯？

這一步很重要——使用者的引導讓 wiki 反映他們的視角，而不只是資料的平鋪直述。

#### 4. 建立來源摘要頁
在 `wiki/` 建立摘要頁。格式見 `references/conventions.md`。

#### 5. 更新實體頁
來源中的人物、組織、產品等——建立新頁或更新既有頁面。

#### 6. 更新概念頁
來源涉及的概念、理論、方法——建立或更新。

#### 7. 標註矛盾
若新資料與既有 wiki 主張衝突，在相關頁面明確標註。這是功能，不是 bug——矛盾的存在本身就是有價值的資訊。

#### 8. 維護交叉引用
使用 `[[wikilink]]` 連結相關頁面。主動在新舊頁面之間建立雙向連結。

#### 9. 更新 index.md
在適當類別下新增或更新條目。

#### 10. 追加 log.md
記錄匯入操作、影響的頁面列表。

### 原則
- **不修改 `raw/` 中的檔案**——它們是不可變的真實來源
- 單一來源可能影響 10-15 個 wiki 頁面，這是正常的
- 偏好一次匯入一個來源，與使用者持續互動
- 批次匯入可行，但監督程度較低

## Query — 查詢知識庫

### 流程

#### 0. 讀取 CLAUDE.md Profile

閱讀 CLAUDE.md 中的知識庫 profile，確認以下設定：
- **知識庫目的與使用者** → 調整回答的深度與語氣
- **組織邏輯** → 影響搜尋相關頁面的策略
- **摘要風格** → 回答風格與 profile 一致

#### 1. 讀取 index.md
掃描索引找出與問題相關的頁面。

#### 2. 讀取相關頁面
深入閱讀找到的 wiki 頁面。在中等規模（約 100 來源、數百頁）下，索引 + 閱讀的方式意外地很有效。

#### 3. 整合回答
綜合多頁資訊，附帶引用（標明來源頁面）。

#### 4. 選擇呈現格式
根據問題性質：
- 一般問題 → Markdown 段落
- 比較 → 表格
- 時間序列 → 時間線
- 複雜分析 → 結構化報告
- 簡報需求 → Marp 格式

#### 5. 詢問是否存入 wiki
好的回答值得保存。比較、分析、新發現的關聯——這些不應消失在聊天紀錄裡。若使用者同意，存為綜整頁或比較頁，更新 index 和 log。

### 回答品質要求
- 引用具體 wiki 頁面，不要泛泛而談
- 標明結論有多少來源支撐
- 指出知識缺口：「目前 wiki 中這方面資料不足」
- 建議可進一步探索的方向

## Lint — 健康檢查

### 檢查項目

1. **矛盾**：不同頁面之間的主張衝突
2. **過時資訊**：已被新來源推翻但未更新的舊主張
3. **孤立頁面**：沒有任何反向連結的頁面
4. **缺漏實體**：被多處提及但沒有獨立頁面的重要概念
5. **缺少交叉引用**：應該互相連結但沒有的頁面
6. **資料缺口**：可透過新來源補充的領域
7. **建議問題**：值得進一步探索的新問題與新來源
8. **Profile 一致性**：頁面是否符合 CLAUDE.md 中的具體指引？包含：
   - 設定「不保留原文引用」但某些頁面有 Key Quotes 區塊
   - 自訂欄位是否在所有適用頁面中一致存在
   - 摘要長度是否符合風格設定
   - 這些不一致可能因 profile 更新後舊頁面未跟上，標出讓使用者決定是否批次修正

### 輸出格式

以清單呈現發現，依嚴重程度排序。每項包含：
- 問題描述
- 涉及的頁面
- 建議的修正方式

等使用者確認後再執行修正——不要自動改東西。

### 修正流程

使用者批准後：
1. 執行修正
2. 更新 index.md
3. 追加 log.md 記錄 lint 操作與修正內容

## 快速參考

### 頁面 Frontmatter

```yaml
---
title: 頁面標題
type: source | entity | concept | synthesis | comparison
sources: [來源1.md, 來源2.md]
created: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [tag1, tag2]
---
```

### 交叉引用格式

- `[[頁面名稱]]`
- `[[頁面名稱|顯示文字]]`
- `[[頁面名稱#標題]]`

### Index 條目格式

```markdown
- [[頁面名稱]] — 一行摘要（來源數：N）
```

### Log 條目格式

```markdown
## [YYYY-MM-DD] 操作類型 | 標題
- 細節
- 影響頁面：[[page1]], [[page2]]
```

完整頁面格式慣例見 `references/conventions.md`。
